## 2. 计算 advantages：group-normalized rewards

### 2.1 思想

对于每个 rollout batch，需要为其中每个样例计算 advantage。这里的 advantage 来自同一个问题下多个 rollout response 的 group-normalized rewards。

设一个问题有一组模型输出：

\[
\{o^{(i)}\}_{i=1}^{G} \sim \pi_\theta(\cdot \mid q),
\]

其中 \(G\) 是每个问题采样的 response 数量。令第 \(i\) 个 response 的 reward 为：

\[
r^{(i)} = R(q,o^{(i)}).
\]

标准 GRPO 使用如下归一化：

\[
A^{(i)} = \frac{r^{(i)} - \operatorname{mean}(r^{(1)},\ldots,r^{(G)})}
{\operatorname{std}(r^{(1)},\ldots,r^{(G)}) + \varepsilon}.
\]

其中 \(\varepsilon\) 是一个小常数，用于避免除以零。

Dr. GRPO 指出，用 group standard deviation 归一化可能会不理想，因为它会给组内正确性变化较小的问题更高权重。于是提出去掉标准差归一化，只保留减去组内均值：

\[
A^{(i)} = r^{(i)} - \operatorname{mean}(r^{(1)},\ldots,r^{(G)}).
\]

本作业要求实现这两种变体，并在后续实验中比较它们的性能。

### 2.2 函数：`compute_group_normalized_rewards`

**任务名：** `compute_group_normalized_rewards`  
**分值：** 2 分  
**目标：** 为每个 rollout response 计算 raw rewards，然后按 group 进行归一化，返回 normalized rewards、raw rewards 和元数据。

推荐接口：

```python
def compute_group_normalized_rewards(
    reward_fn,
    rollout_responses,
    repeated_ground_truths,
    group_size,
    advantage_eps,
    normalize_by_std,
):
    """
    为每个 rollout response 组计算 rewards，并按 group size 归一化。
    """
```

参数说明：

- `reward_fn: Callable[[str, str], dict[str, float]]`  
  根据 ground truth 给 rollout responses 打分，返回包含 `"reward"`、`"format_reward"` 和 `"answer_reward"` 的字典。

- `rollout_responses: list[str]`  
  policy 生成的 rollouts。列表长度为

  \[
  \text{rollout\_batch\_size}
  = \text{n\_prompts\_per\_rollout\_batch} \times \text{group\_size}.
  \]

- `repeated_ground_truths: list[str]`  
  每个样例对应的 ground truth。该列表长度同样为 `rollout_batch_size`，因为每个问题的 ground truth 会重复 `group_size` 次。

- `group_size: int`  
  每个问题对应的 response 数量。

- `advantage_eps: float`  
  防止标准差归一化时除以零的小常数。

- `normalize_by_std: bool`  
  若为 `True`，则除以每组标准差；若为 `False`，则只减去组内均值。

返回值：

```python
tuple[torch.Tensor, torch.Tensor, dict[str, float]]
```

其中：

- `advantages`：形状为 `(rollout_batch_size,)`，表示每个 rollout response 的 group-normalized reward；
- `raw_rewards`：形状为 `(rollout_batch_size,)`，表示未归一化的 raw reward；
- `metadata`：你选择记录的其他统计量，例如 rewards 的 mean、std、max、min。

测试命令：

```bash
uv run pytest -k test_compute_group_normalized_rewards
```

## 3. Naive policy gradient loss

### 3.1 思想

接下来需要实现若干用于 policy gradient 的“loss”。需要注意，这些量并不是通常监督学习意义下的 loss，不应该作为评估指标报告。在 RL 中，更有意义的指标是 train returns、validation returns 以及相关调试指标。

naive policy gradient loss 的形式很直接：把 advantage 与 action 的 log-probability 相乘，并取负号。对于问题 \(q\)、response \(o\) 和 response token \(o_t\)，per-token policy gradient loss 为：

\[
- A_t \log p_\theta(o_t \mid q,o_{<t}).
\]

### 3.2 函数：`compute_naive_policy_gradient_loss`

**任务名：** `compute_naive_policy_gradient_loss`  
**分值：** 1 分  
**目标：** 使用 raw rewards 或已经计算好的 advantages，计算每个 token 的 policy-gradient loss。

推荐接口：

```python
def compute_naive_policy_gradient_loss(
    raw_rewards_or_advantages: torch.Tensor,
    policy_log_probs: torch.Tensor,
) -> torch.Tensor:
    """
    计算每个 token 的 policy-gradient loss。
    raw_rewards_or_advantages 可以是 raw reward，也可以是归一化后的 advantage。
    """
```

参数说明：

- `raw_rewards_or_advantages: torch.Tensor`  
  形状为 `(batch_size, 1)`，表示每个 rollout response 的标量 reward 或 advantage。

- `policy_log_probs: torch.Tensor`  
  形状为 `(batch_size, sequence_length)`，表示每个 token 的 log-probability。

返回值：

- `torch.Tensor`：形状为 `(batch_size, sequence_length)`，表示 per-token policy-gradient loss。训练循环中会继续跨 batch 和 sequence 维度聚合。

实现提示：

- 需要将 `raw_rewards_or_advantages` broadcast 到 `sequence_length` 维度。

测试命令：

```bash
uv run pytest -k test_compute_naive_policy_gradient_loss
```

## 4. GRPO-Clip loss

### 4.1 思想

GRPO-Clip loss 是本节中更核心的 policy-gradient loss。其 per-token 形式为：

\[
-\min\left(
\frac{\pi_\theta(o_t\mid q,o_{<t})}{\pi_{\theta_{\mathrm{old}}}(o_t\mid q,o_{<t})}A_t,
\operatorname{clip}\left(
\frac{\pi_\theta(o_t\mid q,o_{<t})}{\pi_{\theta_{\mathrm{old}}}(o_t\mid q,o_{<t})},
1-\epsilon,
1+\epsilon
\right)A_t
\right).
\]

这里的 ratio 可以用 log-probability 的差来稳定计算：

\[
\operatorname{ratio}_t
= \exp\left(\log \pi_\theta(o_t\mid q,o_{<t})
- \log \pi_{\theta_{\mathrm{old}}}(o_t\mid q,o_{<t})\right).
\]

### 4.2 函数：`compute_grpo_clip_loss`

**任务名：** `compute_grpo_clip_loss`  
**分值：** 2 分  
**目标：** 计算 per-token GRPO-Clip loss。

推荐接口：

```python
def compute_grpo_clip_loss(
    advantages: torch.Tensor,
    policy_log_probs: torch.Tensor,
    old_log_probs: torch.Tensor,
    cliprange: float,
) -> tuple[torch.Tensor, dict[str, torch.Tensor]]:
    """
    计算 per-token GRPO-Clip loss。
    """
```

参数说明：

- `advantages: torch.Tensor`  
  形状为 `(batch_size, 1)`，表示每个样例的 advantage \(A\)。

- `policy_log_probs: torch.Tensor`  
  形状为 `(batch_size, sequence_length)`，来自正在训练的 policy 的 per-token log-probabilities。

- `old_log_probs: torch.Tensor`  
  形状为 `(batch_size, sequence_length)`，来自旧 policy 的 per-token log-probabilities。

- `cliprange: float`  
  clipping 参数 \(\epsilon\)，例如 `0.2`。

返回值：

```python
tuple[torch.Tensor, dict[str, torch.Tensor]]
```

其中：

- `loss`：形状为 `(batch_size, sequence_length)`，表示 per-token clipped loss；
- `metadata`：你希望记录的其他内容。建议记录每个 token 是否被 clipped，也就是 clipped policy-gradient loss 是否低于 unclipped policy-gradient loss。

实现提示：

- 将 `advantages` broadcast 到 `sequence_length` 维度。

测试命令：


## 5. Policy gradient loss wrapper

### 5.1 三种 loss type

作业要求后续运行 ablations，比较三种 policy gradient 版本：

1. `no_baseline`  
   不使用 baseline 的 naive policy gradient loss。此时 advantage 只是 raw reward：

   \[
   A = R(q,o).
   \]

2. `reinforce_with_baseline`  
   使用 group-normalized rewards 作为 advantage 的 naive policy gradient loss。若 \(\bar r\) 是 `compute_group_normalized_rewards` 得到的 group-normalized reward，则：

   \[
   A = \bar r.
   \]

   这里的 group-normalized rewards 可以除以 group standard deviation，也可以不除。

3. `grpo_clip`  
   使用 GRPO-Clip loss。

为了方便切换，需要实现一个统一的 wrapper。

### 5.2 函数：`compute_policy_gradient_loss`

**任务名：** `compute_policy_gradient_loss`  
**分值：** 1 分  
**目标：** 根据 `loss_type` 分发到对应的 loss routine，并返回 per-token loss 和辅助统计量。

推荐接口：

```python
def compute_policy_gradient_loss(
    policy_log_probs: torch.Tensor,
    loss_type: Literal["no_baseline", "reinforce_with_baseline", "grpo_clip"],
    raw_rewards: torch.Tensor | None = None,
    advantages: torch.Tensor | None = None,
    old_log_probs: torch.Tensor | None = None,
    cliprange: float | None = None,
) -> tuple[torch.Tensor, dict[str, torch.Tensor]]:
    """
    选择并计算所需的 policy-gradient loss。
    """
```

参数说明：

- `policy_log_probs: torch.Tensor`  
  形状为 `(batch_size, sequence_length)`，来自正在训练的 policy 的 per-token log-probabilities。

- `loss_type`  
  只能是 `"no_baseline"`、`"reinforce_with_baseline"` 或 `"grpo_clip"` 之一。

- `raw_rewards`  
  当 `loss_type == "no_baseline"` 时必需；形状为 `(batch_size, 1)`。

- `advantages`  
  对 `"reinforce_with_baseline"` 和 `"grpo_clip"` 必需；形状为 `(batch_size, 1)`。

- `old_log_probs`  
  对 `"grpo_clip"` 必需；形状为 `(batch_size, sequence_length)`。

- `cliprange`  
  对 `"grpo_clip"` 必需；表示 clipping 的标量 \(\epsilon\)。

返回值：

```python
tuple[torch.Tensor, dict[str, torch.Tensor]]
```

其中：

- `loss`：形状为 `(batch_size, sequence_length)` 的 per-token loss；
- `metadata`：来自底层 routine 的统计量，例如 GRPO-Clip 的 clip fraction。

实现提示：

- 委托给 `compute_naive_policy_gradient_loss` 或 `compute_grpo_clip_loss`；
- 执行必要的参数检查，可以参考前文的 assertion pattern；
- 将底层 routine 返回的 metadata 聚合到统一字典中。

测试命令：

```bash
uv run pytest -k test_compute_policy_gradient_loss
```

## 6. Masked mean

### 6.1 思想

在前面步骤中，已经可以计算 advantages、log-probabilities、per-token losses，以及 token entropy、clip fraction 等统计量。接下来需要把形状为 `(batch_size, sequence_length)` 的 per-token loss 缩减为每个样例一个标量 loss。

做法是在 sequence 维度上取均值，但只在 response token 对应的位置计算，即只考虑满足：

\[
\operatorname{mask}[i,j] = 1
\]

的 token positions。

按 sequence length 归一化是许多 LLM RL 代码库中的常见做法，但它不一定总是理论上最自然的选择。因为在 policy gradient estimate 中，并没有显式的 \(1/T^{(i)}\) 归一化因子。本节先实现标准 primitive：`masked_mean`。后续实验中会测试使用 SFT 部分实现的 `masked_normalize`。

如果 `dim=None`，则在所有 masked elements 上计算均值。这可用于统计 response tokens 上的 average per-token entropy、clip fraction 等。

### 6.2 函数：`masked_mean`

**任务名：** `masked_mean`  
**分值：** 1 分  
**目标：** 在尊重布尔 mask 的情况下对 tensor 元素求平均。

推荐接口：

```python
def masked_mean(
    tensor: torch.Tensor,
    mask: torch.Tensor,
    dim: int | None = None,
) -> torch.Tensor:
    """
    沿给定维度计算 tensor 的均值，只考虑 mask == 1 的元素。
    """
```

参数说明：

- `tensor: torch.Tensor`  
  要被平均的数据。

- `mask: torch.Tensor`  
  与 `tensor` 形状相同；值为 `1` 的位置会被纳入均值。

- `dim: int | None`  
  计算均值的维度。如果为 `None`，则在所有 masked elements 上计算均值。

返回值：

- `torch.Tensor`：masked mean，形状遵循 `tensor.mean(dim)` 的语义。

测试命令：

```bash
uv run pytest -k test_masked_mean
```

## 7. GRPO microbatch train step

### 7.1 思想

现在可以实现 GRPO 的单个 microbatch train step。若 `gradient_accumulation_steps > 1`，则一个 train minibatch 会被拆分为多个 microbatches。

给定 raw rewards 或 advantages 以及 log-probabilities，训练步骤如下：

1. 计算 per-token policy-gradient loss；
2. 使用 `response_mask` 和 `masked_mean` 把每个样例的 token-level loss 聚合成样例级 loss；
3. 在 batch 维度上取平均；
4. 按 `gradient_accumulation_steps` 调整 loss；
5. 调用 `loss.backward()` 反向传播。

### 7.2 函数：`grpo_microbatch_train_step`

**任务名：** `grpo_microbatch_train_step`  
**分值：** 3 分  
**目标：** 实现 GRPO 的单个 micro-batch update，包括 policy-gradient loss、mask 求平均和 gradient scaling。

推荐接口：

```python
def grpo_microbatch_train_step(
    policy_log_probs: torch.Tensor,
    response_mask: torch.Tensor,
    gradient_accumulation_steps: int,
    loss_type: Literal["no_baseline", "reinforce_with_baseline", "grpo_clip"],
    raw_rewards: torch.Tensor | None = None,
    advantages: torch.Tensor | None = None,
    old_log_probs: torch.Tensor | None = None,
    cliprange: float | None = None,
) -> tuple[torch.Tensor, dict[str, torch.Tensor]]:
    """
    在一个 microbatch 上执行 forward-and-backward pass。
    """
```

参数说明：

- `policy_log_probs: torch.Tensor`  
  形状为 `(batch_size, sequence_length)`，来自正在训练的 policy 的 per-token log-probabilities。

- `response_mask: torch.Tensor`  
  形状为 `(batch_size, sequence_length)`，response token 为 `1`，prompt 和 padding 为 `0`。

- `gradient_accumulation_steps: int`  
  每个 optimizer step 的 microbatches 数。

- `loss_type`  
  `"no_baseline"`、`"reinforce_with_baseline"` 或 `"grpo_clip"` 之一。

- `raw_rewards`  
  当 `loss_type == "no_baseline"` 时需要；形状为 `(batch_size, 1)`。

- `advantages`  
  当 `loss_type != "no_baseline"` 时需要；形状为 `(batch_size, 1)`。

- `old_log_probs`  
  GRPO-Clip 必需；形状为 `(batch_size, sequence_length)`。

- `cliprange`  
  GRPO-Clip 的 clipping 参数 \(\epsilon\)。

返回值：

```python
tuple[torch.Tensor, dict[str, torch.Tensor]]
```

其中：

- `loss`：标量 tensor，表示已经针对 gradient accumulation 调整后的 microbatch loss；
- `metadata`：来自底层 loss 调用的元数据，以及你希望记录的其他统计量。

实现提示：

- 应当在该函数中调用 `loss.backward()`；
- 注意根据 `gradient_accumulation_steps` 对 loss 做缩放。

测试命令：

```bash
uv run pytest -k test_grpo_microbatch_train_step
```

## 8. 整合：GRPO train loop

### 8.1 任务目标

现在需要把完整 GRPO train loop 组合起来。整体结构应参考 §7.1 的 GRPO 算法，并复用已经实现的函数。

**任务名：** `grpo_train_loop`  
**分值：** 5 分  
**目标：** 实现完整的 GRPO train loop，在 MATH 上训练 policy，确认 validation rewards 上升，并检查 rollouts 是否随训练合理变化。

交付物：

- validation rewards 随 steps 变化的 plot；
- 训练过程中的若干 example rollouts。

### 8.2 Starter hyperparameters

如果实现正确，使用以下参数应能看到合理结果：

```python
n_grpo_steps: int = 200
learning_rate: float = 1e-5
advantage_eps: float = 1e-6
rollout_batch_size: int = 256
group_size: int = 8
sampling_temperature: float = 1.0
sampling_min_tokens: int = 4  # As in Expiter, disallow empty string responses
sampling_max_tokens: int = 1024
epochs_per_rollout_batch: int = 1  # On-policy
train_batch_size: int = 256  # On-policy
gradient_accumulation_steps: int = 128  # microbatch size is 2, will fit on H100
gpu_memory_utilization: float = 0.85
loss_type: Literal[
    "no_baseline",
    "reinforce_with_baseline",
    "grpo_clip",
] = "reinforce_with_baseline"
use_std_normalization: bool = True

optimizer = torch.optim.AdamW(
    policy.parameters(),
    lr=learning_rate,
    weight_decay=0.0,
    betas=(0.9, 0.95),
)
```

这些默认超参数对应 on-policy 设置：对每个 rollout batch，只执行一个 gradient step。等价地：

```python
train_batch_size == rollout_batch_size
epochs_per_rollout_batch == 1
```

### 8.3 Sanity check asserts 与派生常量

下面这些检查可以避免边界情况，并帮助确认训练循环设置合理：

```python
assert train_batch_size % gradient_accumulation_steps == 0, (
    "train_batch_size must be divisible by gradient_accumulation_steps"
)
micro_train_batch_size = train_batch_size // gradient_accumulation_steps

assert rollout_batch_size % group_size == 0, (
    "rollout_batch_size must be divisible by group_size"
)
n_prompts_per_rollout_batch = rollout_batch_size // group_size

assert train_batch_size >= group_size, (
    "train_batch_size must be greater than or equal to group_size"
)

n_microbatches_per_rollout_batch = rollout_batch_size // micro_train_batch_size
```

### 8.4 训练循环中的提示

实现 GRPO train loop 时，需要注意以下事项：

1. 使用 `r1_zero` prompt。
2. 指示 vLLM 在第二个 answer tag `</answer>` 处停止 generation，与前面实验保持一致。
3. 建议使用 `typer` 做参数解析。
4. 使用 gradient clipping，clip value 为 `1.0`。
5. 应当定期记录 validation rewards，例如每 5 或 10 步记录一次。
6. 为了比较超参数，至少在 1024 个 validation examples 上评估，因为 CoT/RL evaluations 通常比较 noisy。
7. 按照本作业的 loss 实现，GRPO-Clip 应只在 off-policy 时使用，因为它需要 `old_log_probs`。
8. 在多个 epochs 的 off-policy 设置中，不要为每个 epoch 重新计算 `old_log_probs`。应当在每次 rollout batch generation phase 后计算一次，并在每个 epoch 中复用。
9. 不应对 `old_log_probs` 求导。
10. 每次 optimizer update 时，建议记录以下内容：
    - loss；
    - gradient norm；
    - token entropy；
    - 如果是 off-policy，记录 clip fraction；
    - train rewards，包括 total、format 和 answer；
    - 其他有助于调试的指标。

## 10. 最小实现逻辑图

```text
rollout prompts
    ↓
vLLM 生成 group_size 个 responses
    ↓
reward_fn 计算 raw rewards
    ↓
compute_group_normalized_rewards 得到 advantages
    ↓
tokenize prompt + response
    ↓
policy 计算 policy_log_probs
    ↓
若 off-policy：旧 policy 计算 old_log_probs，且不求导
    ↓
compute_policy_gradient_loss
    ↓
masked_mean 按 response_mask 聚合 token-level loss
    ↓
按 gradient_accumulation_steps 缩放
    ↓
loss.backward()
    ↓
optimizer.step(), optimizer.zero_grad()
    ↓
定期评估 validation rewards，并记录 rollouts
```
